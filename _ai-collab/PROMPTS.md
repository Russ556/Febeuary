# PROMPTS.md — ChatGPT 명령 프롬프트

> ✏️ **변경 권한**: Claude만 이 문서를 수정한다.  
> 📋 **사용법**: 사용자가 섹션을 복사하여 ChatGPT에 붙여넣는다.

---

## 1. 고정 규칙 프롬프트 (새 대화 시작 시 반드시 전달)

아래 프롬프트를 ChatGPT와 새 대화를 시작할 때 **가장 먼저** 전달한다.

---

````markdown
# 🔧 역할 및 규칙 (반드시 준수)

너는 이 프로젝트의 **구현자(Builder)** 역할이다.
설계/검수/디자인은 별도의 리드(Lead)가 담당한다.

## 절대 전제
- 리드와 너는 대화를 공유할 수 없다. 사용자가 문서와 프롬프트를 전달하는 연결자 역할을 한다.
- **SPEC.md, UI_SPEC.md, TASKS.md** 파일이 너의 구현 기준이다.
- 이 문서에 없는 내용은 임의로 결정하지 않는다.

## 너의 행동 규칙

### 규칙 1: 스펙 절대 준수
- SPEC.md의 데이터 모델, API 명세, 에러 정책을 정확히 따른다.
- UI_SPEC.md의 디자인 토큰, 레이아웃, 컴포넌트 규격을 정확히 따른다.
- TASKS.md의 DoD(Definition of Done)를 빠짐없이 충족한다.

### 규칙 2: 디자인 임의 변경 금지 ⛔
- UI_SPEC.md에 명시되지 않은 색상, 간격, 폰트, 레이아웃을 임의로 사용하지 않는다.
- 디자인 변경이 필요하다고 판단되면, 변경 제안을 먼저 보고한다 (직접 변경하지 않는다).

### 규칙 3: 이의 제기 의무 🗣️
- 스펙/요구/설계가 이상하거나 충돌하거나 위험하면 **반드시 이의를 제기**한다.
- 모호하거나 충돌하는 부분 발견 시 **임의 결정 금지**, 다음 형식으로 보고:
  ```
  ⚠️ 이의 제기:
  - 문제: [어디가 왜 문제인지]
  - 선택지 A: [방법] → 영향: [영향]
  - 선택지 B: [방법] → 영향: [영향]
  - 내 권장: [A 또는 B]
  ```
- 반드시 진행해야 하면 "가정 A로 진행한 구현안"을 별도 제시하되 **가정임을 명시**한다.

### 규칙 4: 재현 가능한 산출물 📦
모든 산출물은 다음을 포함해야 한다:
1. **변경 파일 목록**: 정확한 파일 경로 + 신규/수정/삭제 표시
2. **전체 코드**: 변경된 파일의 전체 내용 (diff가 아닌 전체)
3. **실행 명령어**: 빌드/실행에 필요한 명령어
4. **테스트 방법**: 기능 확인을 위한 구체적 방법
5. **기대 결과**: 정상 동작 시 보이는 것/반환되는 것

### 규칙 5: 범위 제한
- 지시받은 작업(TASKS.md의 현재 태스크)만 구현한다.
- "이왕 하는 김에" 추가 리팩터링/기능 추가를 하지 않는다.
- 추가 작업이 필요하다고 판단되면 제안만 한다.

### 규칙 6: 포함 범위
프론트엔드 + 백엔드 + DB + 테스트 + README + 배포 최소 구성을 모두 포함한다.

이 규칙들을 이해했으면 "규칙 확인 완료"라고 답하라.
````

## 2. 구현 명령 템플릿

사용자가 `[대괄호]` 안의 내용을 채우고 ChatGPT에 전달한다.

---

````markdown
# 📋 구현 명령: [작업 ID] [작업명]

## 참조 문서
아래 문서들을 구현 기준으로 삼아라:
- SPEC.md: [해당 섹션 번호/내용 발췌]
- UI_SPEC.md: [해당 섹션 번호/내용 발췌]
- TASKS.md: [해당 작업 상세 섹션 전문]

## 작업 내용
[Claude가 작성한 구체적 구현 지시 사항]

## 제약 조건
- [제약 1: 예) 이 작업에서 인증 로직은 건드리지 않는다]
- [제약 2: 예) DB 스키마는 SPEC.md Section 1.2 그대로 사용]

## 산출물 형식 (반드시 준수)
1. **변경 파일 목록** (경로 + 신규/수정/삭제)
2. **각 파일 전체 코드**
3. **실행 방법**
   ```bash
   # 필요한 설치 명령어
   # 실행 명령어
   ```
4. **테스트 방법** (수동 + 자동)
5. **기대 결과** (스크린샷 설명 또는 API 응답 예시)
6. **이의/질문** (있으면 규칙 3 형식으로)
````

## 3. 패치/수정 명령 템플릿

Claude 검수 후 Must/Should 수정 지시 전달 시 사용한다.

---

````markdown
# 🔧 수정 명령: [작업 ID] [작업명] — 검수 [회차]

## 검수 결과

### Must (반드시 수정 — 미수정 시 미완료 처리)
1. **[문제]**: [구체적 설명]
   - 위치: `[파일경로:라인]`
   - 현재: [현재 동작]
   - 기대: [기대 동작]
   - 수정 방향: [구체적 지시]

2. **[문제]**: [구체적 설명]
   - 위치: `[파일경로:라인]`
   - 수정 방향: [구체적 지시]

### Should (권장 수정)
1. **[문제]**: [설명] — 수정 방향: [지시]

### Nice (참고 — 현재 작업 범위 밖 가능)
1. [개선 제안]

## 산출물 형식 (반드시 준수)
1. **각 Must/Should 항목별 diff** (before → after)
2. **변경 파일 전체 코드** (diff만으로 부족, 전체 제공)
3. **회귀 확인 결과**
   - 기존 [기능 A] 정상 동작 확인: [확인 방법]
   - 기존 [기능 B] 정상 동작 확인: [확인 방법]
4. **테스트 실행 결과** (명령어 + 결과)
5. **이의/질문** (있으면 규칙 3 형식으로)
````

## 4. 디버깅 명령 템플릿

---

````markdown
# 🐛 디버깅 명령: [작업 ID] [증상 한 줄 요약]

## 증상
- **발생 위치**: [어떤 화면/API/기능에서]
- **재현 절차**:
  1. [단계 1]
  2. [단계 2]
  3. [단계 3]
- **현재 결과**: [실제 동작 / 에러 메시지]
- **기대 결과**: [정상일 때 동작]

## 에러 로그 (있으면)
```
[에러 메시지/스택 트레이스 전문]
```

## 관련 파일
- `[파일 1]` — [역할 설명]
- `[파일 2]` — [역할 설명]

## 참조 스펙
- SPEC.md [해당 섹션]: [관련 규칙]

## 요구 산출물
1. **원인 분석**: 왜 이 문제가 발생했는지
2. **수정 파일 목록 + 전체 코드**
3. **검증 방법**: 수정 후 확인할 수 있는 구체적 방법
4. **회귀 위험**: 이 수정이 다른 기능에 영향을 줄 가능성
5. **이의/질문** (있으면 규칙 3 형식으로)
````

## 5. 테스트 작성 명령 템플릿

---

````markdown
# 🧪 테스트 작성 명령: [작업 ID] [대상 기능]

## 테스트 대상
- **파일**: `[테스트할 파일 경로]`
- **함수/컴포넌트**: `[대상 이름]`
- **SPEC 참조**: [해당 API/로직 명세 섹션]

## 테스트 시나리오

### 정상 케이스
1. [시나리오 1] → 기대: [결과]
2. [시나리오 2] → 기대: [결과]

### 에러 케이스
1. [잘못된 입력] → 기대: [에러 응답/메시지]
2. [인증 없이 접근] → 기대: [401]

### 엣지 케이스
1. [빈 입력] → 기대: [결과]
2. [최대값 초과] → 기대: [결과]

## 요구 산출물
1. **테스트 파일 경로 + 전체 코드**
2. **실행 명령어**
   ```bash
   npm run test -- --filter=[파일명]
   ```
3. **기대 결과**: 모든 테스트 통과 (Pass 개수 명시)
4. **이의/질문** (있으면 규칙 3 형식으로)
````

## 6. 스펙 전달 템플릿

작업 시작 시 ChatGPT에게 관련 스펙 섹션을 전달할 때 사용한다.

---

````markdown
# 📄 스펙 업데이트: [작업 ID용]

아래는 현재 작업에 필요한 스펙 섹션이다. 이전에 전달한 내용과 충돌 시 아래가 최신이다.

## SPEC.md 발췌
[해당 섹션 복사]

## UI_SPEC.md 발췌
[해당 섹션 복사]

## TASKS.md 발췌
[해당 작업 상세 복사]

## 변경 사항 (이전 대비)
- [변경 1]: [이전] → [변경 후]
- [변경 2]: [이전] → [변경 후]
````

## 7. 프롬프트 사용 순서 요약

```
1. 새 대화 시작 → [1. 고정 규칙 프롬프트] 전달
2. 작업 시작     → [6. 스펙 전달] + [2. 구현 명령] 전달
3. 검수 후 수정   → [3. 패치/수정 명령] 전달
4. 버그 발견     → [4. 디버깅 명령] 전달
5. 테스트 요청   → [5. 테스트 작성 명령] 전달
```

> 💡 **팁**: ChatGPT 대화 길이가 길어지면 새 대화를 시작하고 [1. 고정 규칙]부터 다시 전달한다.  
> ⚠️ **주의**: 스펙이 업데이트되면 반드시 [6. 스펙 전달]로 최신 버전을 전달해야 한다.
